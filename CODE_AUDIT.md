Область: Flask API, RAG/Chroma, история чатов, загрузка документов, frontend, конфигурация и тесты.

## Краткий вывод

Проект рабочий по основным сценариям: текущий тестовый набор проходит (`24 passed`). Основные риски находятся не в RAG-логике, а вокруг эксплуатационной безопасности: доступ к API по умолчанию открыт, destructive операции доступны через обычные endpoint'ы, лимиты загрузки файлов не применяются, а переиндексация может удалить текущую Chroma-коллекцию до успешной сборки новой.

Самые приоритетные зоны для исправления:

1. Закрыть API и админские/destructive операции обязательной авторизацией.
2. Убрать API-ключи из query string и не разрешать CORS `*` в production.
3. Ввести реальные backend-лимиты для upload, `top_k`, `min_score`, размера JSON и числа фоновых задач.
4. Сделать переиндексацию атомарной: не удалять рабочую коллекцию до успешной сборки новой.
5. Усилить безопасное открытие загруженных документов и Markdown-рендеринг.

## Находки

### High: API по умолчанию может быть полностью открыт

Файлы: `web_app.py`, `config/settings.py`, `api/routes/chat.py`, `api/routes/documents.py`, `api/routes/admin.py`

В `config/settings.py` `API_KEY` и `ADMIN_API_KEY` по умолчанию пустые. Проверка в `web_app.py` включается только если задан `settings.API_KEY`, а админский ключ проверяется только внутри этой ветки. Поэтому при дефолтной конфигурации открыты все `/api/*`, включая:

- `DELETE /api/chats` - удалить всю историю чатов.
- `POST /api/documents/upload` - загрузить документ.
- `POST /api/documents/reindex` - запустить переиндексацию.
- `GET /api/admin/overview` - получить runtime-настройки, пути, модели и диагностические данные.

Дополнительно `CORS_ORIGINS` по умолчанию равен `*`, а приложение слушает `0.0.0.0`, что повышает риск случайной публикации API в сеть.

Рекомендация: сделать авторизацию обязательной для production, разделить обычный API и admin/destructive права, блокировать запуск production с пустыми ключами и `CORS_ORIGINS=*`.

### High: API-ключи принимаются через query string

Файл: `web_app.py`

В `before_request` ключи берутся не только из заголовков `X-API-Key` / `X-Admin-Key`, но и из параметров `api_key` / `admin_key`. Query string часто попадает в историю браузера, access logs, reverse proxy logs, referrer headers и скриншоты.

Рекомендация: принимать ключи только из заголовков, добавить ротацию ключей и явно запретить query-string аутентификацию.

### High: Переиндексация удаляет рабочую Chroma-коллекцию до успешного rebuild

Файлы: `api/routes/documents.py`, `create_vector_db.py`, `web_app.py`

`POST /api/documents/reindex` запускает `_run_reindex()`, а `create_vector_db()` сначала делает `client.delete_collection(settings.CHROMA_COLLECTION_NAME)`, затем заново собирает эмбеддинги и сохраняет документы. Если эмбеддинги, LLM-сервер, файловый парсинг или Chroma упадут после удаления, рабочая база будет потеряна или останется пустой.

Есть второй риск: `web_app.py` держит глобальный `rag_system` и `db_initialized=True`. После переиндексации объект может продолжить использовать старую/удаленную коллекцию, пока приложение не перезапустят.

Рекомендация: строить новую коллекцию под временным именем, проверять count/качество, затем атомарно переключать активную коллекцию. После успешной переиндексации сбрасывать/пересоздавать `rag_system`.

### High: `MAX_FILE_SIZE` объявлен, но не применяется

Файлы: `config/settings.py`, `api/routes/documents.py`

`settings.MAX_FILE_SIZE` задан, но Flask `MAX_CONTENT_LENGTH` не настроен, а `upload_document()` и `preview_document_upload()` не проверяют размер файла перед сохранением/парсингом. Большой PDF/DOCX/XLSX может занять диск, память или поток обработки.

Рекомендация: выставить `app.config["MAX_CONTENT_LENGTH"]`, проверить `Content-Length`, проверять фактический размер временного файла и добавить тесты на превышение лимита.

### High: Открытие загруженных HTML-документов выполняется в origin приложения

Файлы: `api/routes/documents.py`, `templates/index.html`, `static/script.js`

`/api/documents/open` отдает исходный файл через `send_file(..., as_attachment=False)`. Если в базе знаний есть HTML/HTM, браузер откроет его в том же origin, что и приложение. Загруженный HTML может содержать JavaScript и выполнять запросы к API как страница приложения, особенно опасно при открытом API/CORS.

Рекомендация: для HTML отдавать файл как attachment или через отдельный sandboxed origin; добавить CSP (`sandbox`, запрет script) или безопасный текстовый preview вместо открытия исходного HTML.

### Medium: Клиентские ограничения `top_k` и `min_score` не дублируются на backend

Файлы: `web_app.py`, `core/rag.py`, `static/script.js`

UI задает `top_k` в диапазоне 1..10 и `min_score` 0..1, но API принимает произвольные значения. `top_k` напрямую идет в Chroma `n_results`, а `min_score` участвует в фильтрации. Большой `top_k` может создать лишнюю нагрузку на Chroma и LLM-контекст, отрицательный `min_score` фактически отключает фильтр.

Рекомендация: валидировать и clamp'ить параметры на сервере; добавить тесты на `top_k=100000`, `top_k=0`, `min_score=-1`, `min_score=2`.

### Medium: Upload перезаписывает существующий файл без явного подтверждения

Файл: `api/routes/documents.py`

`upload_document()` сохраняет файл в `UPLOAD_DIR / secure_filename(file.filename)`. Если файл с таким именем уже существует, он будет перезаписан. Preview умеет находить duplicate и показывать diff, но сам upload не требует подтверждения версии.

Рекомендация: запрещать overwrite по умолчанию, добавлять версионирование имени или требовать явный флаг `overwrite=true` после preview.

### Medium: Параметры чанкования из env могут привести к бесконечному циклу

Файл: `create_vector_db.py`

`chunk_text()` делает `start = end - overlap`. Если `CHUNK_OVERLAP >= CHUNK_SIZE` или `CHUNK_SIZE <= 0`, прогресс цикла ломается. Эти значения читаются из env без проверки.

Рекомендация: валидировать инвариант `0 <= CHUNK_OVERLAP < CHUNK_SIZE`, а также разумные границы `CHUNK_SIZE`.

### Medium: Логи могут содержать чувствительные пользовательские данные

Файлы: `web_app.py`, `core/rag.py`, `config/logging_config.py`

В логах пишутся фрагменты запросов, тела JSON, prompt/RAG-диагностика и ошибки. Для корпоративной базы знаний это может включать персональные данные, внутренние документы, токены, коммерческую информацию. Сейчас маскируются только ключи `password`, `token`, `secret`, но не `api_key`, `admin_key`, `Authorization`, текст вопроса или содержимое источников.

Рекомендация: в production понизить детализацию, маскировать ключи и заголовки, не логировать полный текст запросов/промптов, настроить срок хранения логов.

### Medium: Markdown/XSS-защита зависит от CDN и fallback-санаитайзер слабее DOMPurify

Файлы: `templates/index.html`, `static/script.js`

Ответы модели рендерятся через `marked`, затем очищаются DOMPurify. Это хороший основной путь, но обе библиотеки грузятся с CDN без SRI. Если DOMPurify не загрузится, используется самописный `sanitizeHtml()`, который удаляет часть опасных тегов/атрибутов, но не является полноценным HTML sanitizer'ом.

Рекомендация: поставлять зависимости локально или добавить SRI/CSP; не использовать самописный sanitizer как security fallback, а при отсутствии DOMPurify рендерить plain text.

### Medium: Файловый cache использует `pickle`

Файл: `utils/cache.py`

`FileCache.get()` читает cache-файлы через `pickle.load()`. Если злоумышленник получает возможность записать файл в `CACHE_DIR`, следующий cache hit может привести к выполнению кода в процессе приложения.

Рекомендация: хранить эмбеддинги в JSON/msgpack/numpy-формате без исполнения кода или жестко ограничить права на `CACHE_DIR` и не читать cache-файлы из недоверенных директорий.

### Medium: Фоновые jobs переиндексации живут только в памяти и не ограничены

Файл: `api/routes/documents.py`

`_jobs` - глобальный dict без TTL и без ограничения количества, а каждый `POST /api/documents/reindex` создает новый daemon thread. При повторных запросах можно запустить несколько одновременных reindex-процессов, которые конкурируют за Chroma, LLM и файловую систему.

Рекомендация: разрешить только один reindex одновременно, хранить состояние в SQLite/очереди задач, ограничить историю jobs и добавить rate limit.

### Low: Runtime-настройки раскрывают внутренние детали

Файлы: `api/routes/admin.py`, `web_app.py`

`/api/admin/settings` и `/api/admin/overview` возвращают пути к директориям, имена моделей, CORS-настройки, параметры RAG и Chroma. Секреты не возвращаются, но при открытой авторизации это облегчает разведку и эксплуатацию.

Рекомендация: закрыть admin endpoints отдельным обязательным ключом/ролью и минимизировать публичную диагностическую выдачу.

### Low: Дублирующаяся настройка логгера RAG

Файл: `core/rag.py`

При импорте модуль добавляет `RotatingFileHandler` в логгер `rag`. При reload'ах, тестах или повторной инициализации возможны дублирующиеся handler'ы и повторяющиеся строки логов.

Рекомендация: перед `addHandler` проверять, что аналогичный handler еще не добавлен.

### Low: README и код расходятся по версиям/командам

Файлы: `README.md`, `requirements.txt`

README заявляет Python 3.8+, но проект сейчас запускается на Python 3.14, а зависимости включают свежие версии с deprecation warning'ами. Также README упоминает `.env.example` и `start.bat`, но в текущем дереве аудита эти файлы не были найдены.

Рекомендация: актуализировать README под фактическую структуру, поддерживаемые версии Python и способ запуска.

## Пробелы в тестах

Текущие тесты хорошо покрывают базовые API-сценарии, но почти не проверяют защитные свойства:

- Нет тестов на обязательность `ADMIN_API_KEY` для `/api/admin/*` и destructive endpoints.
- Нет тестов, что API-ключи не принимаются через query string.
- Нет тестов на upload больше `MAX_FILE_SIZE`.
- Нет тестов на overwrite файла при upload.
- Нет тестов на одновременные `/api/documents/reindex`.
- Нет тестов на backend-валидацию `top_k`, `min_score`, `answer_mode`.
- Нет тестов на безопасную отдачу HTML через `/api/documents/open`.
- Нет тестов на плохие env-инварианты `CHUNK_OVERLAP >= CHUNK_SIZE`.
- Нет интеграционного теста, что после reindex `rag_system` переинициализируется.

## Рекомендуемый порядок работ

1. Сначала закрыть доступ: mandatory auth для production, отдельный admin key/role, убрать query string keys, ограничить CORS.
2. Затем защитить данные: безопасная переиндексация, запрет overwrite без подтверждения, лимит upload.
3. Затем защитить UI: sandbox/attachment для HTML, локальные JS-зависимости или SRI/CSP, no-DOMPurify fallback в plain text.
4. После этого расширить тесты на негативные сценарии и добавить CI-команду `python -m pytest`.
