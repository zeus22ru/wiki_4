# Идея: вывод изображений из базы знаний в ответах RAG

## Задача

В базе знаний могут быть документы с изображениями. Нужно дать ассистенту возможность при необходимости показывать релевантные картинки вместе с текстовым ответом.

## Текущее состояние

- Индексация в `create_vector_db.py` извлекает только текст из HTML, DOCX, PDF, XLSX, XLS, PPTX и DOC.
- В ChromaDB сохраняются текстовые чанки и метаданные: `title`, `source`, `path`, `file_type`, `chunk_index`, `total_chunks`.
- API ответа в `web_app.py` возвращает `answer`, `sources`, `citations`, `diagnostics`.
- Фронтенд в `static/script.js` уже умеет показывать Markdown-ответ, источники и открывать исходный документ через `/api/documents/open`.
- Отдельного поля для изображений и эндпоинта безопасной выдачи image assets пока нет.

## Предлагаемый подход

Не заставлять модель вставлять картинки напрямую в Markdown. Лучше передавать изображения как структурированные вложения к найденным источникам:

```json
{
  "answer": "...",
  "sources": [...],
  "citations": [...],
  "images": [
    {
      "id": "source_chunk_0_img_1",
      "url": "/api/documents/images/source_chunk_0_img_1",
      "title": "Схема подключения",
      "caption": "Рисунок из раздела установки",
      "source_path": "manuals/install.docx",
      "file_type": "docx",
      "chunk_id": "...",
      "page": null,
      "slide": null
    }
  ]
}
```

## Этапы реализации

1. Добавить извлечение изображений при индексации.
   - HTML: читать `<img src>`, `alt`, подписи рядом с картинкой.
   - DOCX: извлекать embedded media из пакета документа.
   - PPTX: извлекать картинки со слайдов и номер слайда.
   - PDF: отдельный этап, потому что нужно аккуратно сопоставлять изображения со страницами и текстом.

2. Сохранять image assets в контролируемый каталог внутри проекта, например `data/extracted_images/`.

3. Добавлять к чанкам метаданные о связанных изображениях:
   - `image_ids`
   - `image_captions`
   - `image_pages` или `image_slides`
   - `source_path`

4. Расширить `RAGResult` и API-ответ:
   - добавить `images` на верхнем уровне или `sources[].images`;
   - возвращать только изображения, связанные с найденными релевантными чанками;
   - ограничивать количество изображений, например 3-5 на ответ.

5. Добавить безопасный эндпоинт выдачи изображений.
   - Разрешать только файлы из каталога extracted images.
   - Не принимать произвольные абсолютные пути.
   - Проверять расширение и MIME type.

6. Обновить фронтенд.
   - Показать блок "Изображения из источников" под ответом.
   - Добавить миниатюры, подписи и кнопку открытия оригинального документа.
   - При необходимости показывать изображения в панели источников.

## Логика показа

Первый вариант можно сделать простым: если найденный чанк содержит связанные изображения, показывать их в ответе.

Позже можно улучшить:

- показывать картинки только если вопрос содержит слова вроде "схема", "рисунок", "скриншот", "как выглядит", "пример";
- просить LLM выбрать релевантные изображения из списка кандидатов;
- использовать caption/alt для ранжирования изображений.

## Риски и ограничения

- PDF сложнее HTML/DOCX/PPTX: изображения надо связывать со страницами и текстом, иначе можно показать нерелевантную картинку.
- Нужно не отдавать произвольные файлы с диска через API.
- Нужно продумать размер хранилища extracted images и очистку при переиндексации.
- Если изображение содержит важный текст, обычная текстовая индексация его не увидит. Для этого позже может понадобиться OCR.

## Минимальный полезный MVP

MVP можно ограничить HTML, DOCX и PPTX:

- извлекать изображения в `data/extracted_images/`;
- сохранять `images` в metadata чанка;
- возвращать `images` в `/api/chat` и `/api/chat/stream`;
- отображать изображения под ответом и в источниках.
