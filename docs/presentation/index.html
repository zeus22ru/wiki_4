Коротко: вы отправляете текст вопроса — дальше это уже не «разговор с интернетом». Система ищет ответы в ваших документах, собирает из них контекст и только затем просит языковую модель сформулировать ответ человеческим языком. Ниже — шаги по порядку.

  1. Вопрос из браузера. Вы набираете запрос в чате. Если это продолжение темы, вместе с вопросом учитывается недавняя переписка — чтобы «а для админа?» понималось в контексте прошлого сообщения.
  2. Подготовка к поиску. Система может перефразировать вопрос в ясный поисковый запрос и добавить ещё одну-две формулировки: так проще найти нужный фрагмент, даже если в документе другие слова.
  3. Два поиска и объединение. Параллельно ищем по смыслу (векторный поиск по ChromaDB — «похожие по смыслу куски») и по словам (BM25 — как поиск по точным терминам в тексте). Списки кандидатов склеивают в один рейтинг; при необходимости верхние варианты дополнительно упорядочивают (cross-encoder).
  4. Фрагменты документов. В модель уходит не абы что, а конкретные отрывки из базы знаний — ровно те, что прошли отбор. Это и есть опора ответа: не «память модели из интернета», а ваши регламенты и инструкции.
  5. Ответ модели. Локальная LLM (Ollama или LM Studio) получает вопрос и найденные отрывки и формулирует связный ответ: объяснение, шаги, списки — но опираясь на переданный материал.
  6. Экран для пользователя. Вы видите текст ответа; можно открыть список источников и цитаты — проверить каждую мысль по исходному файлу или странице wiki.
Кратко: пользователь пишет вопрос, мы переписываем его в самодостаточный поисковый запрос, ищем параллельно по векторам и BM25, объединяем через RRF, при необходимости переранжируем, кладём контекст в промпт LLM, отдаём ответ обратно с источниками.

Гибридный поиск

Одной технологии поиска недостаточно: кто-то формулирует как в регламенте, кто-то — своими словами. Запрос идёт двумя дорожками, результаты склеиваются — так мы не теряем ни точное слово, ни близкий по смыслу абзац. Ниже — цепочка шагов, коротко «кто за что отвечает», примеры и итог для модели.

Запрос → BM25 + Dense → RRF → Rerank → Top-K
  • По словам (BM25) — как умный Ctrl+F: «журнал регистрации», «ЭЦП», «ТСД», артикулы.
  • По смыслу (Chroma) — другие формулировки ведут к тому же регламенту («лог событий», «массовая рассылка должникам»).
  • RRF — два топа кандидатов объединяем в один рейтинг: не «или слова, или смысл», а оба учтены.
  • Rerank (опционально) — вторая модель пересортировывает кандидатов по релевантности к вопросу.
Пример 1 — термины 1С

«Ошибка в журнале регистрации» — BM25 цепляет дословно; если в wiki «просмотр событий», помогает поиск по смыслу.

Пример 2 — «живой» вопрос

«Рассылка должникам» vs «информирование о задолженности» в документе — смысл тянет нужный блок; BM25 подхватывает слова «должник», «рассылка», если они есть в тексте.

Итог для модели

В промпт попадают лучшие отрывки из вашей базы после слияния поисков и (если включено) rerank — не «интернет из головы», а ваши документы.

.env: RETRIEVAL_MODE, RRF_K_CONSTANT, RERANK_ENABLED, RERANK_MODEL · core/retrieval.py

Акцент для аудитории: гибрид — это про качество на реальных формулировках сотрудников. Пример журнала регистрации — про BM25; пример рассылки должникам — про смысл + слова.

Структурное чанкирование

Документ нарезаем не «через каждые 500 символов», а по структуре: заголовки, списки, таблицы. К каждому чанку добавляем контекст раздела.

  • HTML — обход DOM по h1–h6, ul/ol, table.
  • DOCX — стили Heading*, таблицы по строкам.
  • В метаданных — section_path, chunk_kind, путь раздела видно в источниках.
  • Contextual Retrieval: LLM добавляет к чанку короткую аннотацию-префикс перед эмбеддингом — улучшает поиск по фрагментам без явного контекста.
Пример Регламент 1С на 40 страниц делится по разделам — ассистент находит именно пункт «Раздача прав на конкретный документ», а не возвращает весь документ или невнятный кусок из середины.

Память диалога

Уточняющие вопросы «а для админа?» работают, потому что мы переписываем запрос с учётом истории, добавляем альтернативные формулировки и гипотетический ответ (HyDE).

  • Rewrite — преобразуем уточнение в один самодостаточный запрос для поиска.
  • Multi-query — генерируем 2–3 альтернативные формулировки, объединяем результаты.
  • HyDE — короткий «гипотетический ответ» как дополнительный dense-запрос.
  • Переписанный запрос сохраняется в SQLite (retrieval_query_text) — можно отлаживать.
Сценарий Пользователь: «Как добавить пользователя в 1С?» →
Ассистент отвечает →
Пользователь: «А как для роли админа?» →
Система переписывает: «Как добавить пользователя 1С с ролью администратор» и находит правильный раздел.
